SECTION 1 — SYSTEM OVERVIEW
[Useful for: Architecture Diagrams, Context Diagrams, Use-Case Diagrams]
================================================================================

Application Name : Horaistion
Type             : Full-Stack Web Application (MERN Stack)
Purpose          : A media catalog platform for browsing Anime, Movies, and
                   TV Shows. Users can explore media, read details, watch
                   trailers, manage a personal watchlist, and write reviews.
                   Admins can add, edit, and delete content via a dashboard.

Actors:
  1. Guest (Unauthenticated)  — Can only access /login and /Register pages.
  2. Registered User          — Can browse media, manage watchlist, write
                                reviews, and update their profile.
  3. Admin User               — Has all User permissions PLUS can access the
                                Admin Dashboard, Add Media, and Edit Media.
  4. Express.js Server        — The backend middleware between browser and DB.
  5. MongoDB Database         — Persistent data store for all collections.

External Systems:
  - YouTube (embedded iframes for trailers — no API call, just URL rendering)
  - Browser localStorage      — Stores JWT token, user email, role, fullname.

SECTION 2 — TECHNOLOGY STACK
[Useful for: Architecture Diagrams, Component Diagrams]
================================================================================

FRONTEND (Client):
  - Framework   : React.js 19 (built with Vite)
  - Routing     : React Router DOM v6 (BrowserRouter)
  - Styling     : Tailwind CSS + Material Tailwind components
  - HTTP Client : Axios (with request/response interceptors)
  - State       : React Hooks only (useState, useEffect) — no Redux
  - Context     : ToastContext (global notification system)
  - Entry Point : main.jsx → App.jsx

BACKEND (Server):
  - Runtime     : Node.js
  - Framework   : Express.js
  - Auth        : JSON Web Tokens (jsonwebtoken) + bcryptjs for hashing
  - DB ODM      : Mongoose
  - Middleware  : cors, express.json(), custom auth/adminAuth guards
  - Port        : 6002

DATABASE:
  - Engine      : MongoDB (local instance, port 27017)
  - DB Name     : maindata
  - Collections : media, mediaMovie, Signup, reviews

SECTION 7 — AUTHENTICATION & AUTHORIZATION FLOW
[Useful for: Sequence Diagrams, Flowcharts, Security Diagrams]
================================================================================

REGISTRATION FLOW:
  1. User fills Registerpage form (fullname, email, password).
  2. Frontend POSTs to /Signup.
  3. Backend checks if email already exists in userModel.
  4. If new → bcrypt hashes the password (salt rounds: 10).
  5. If email contains 'admin' → role set to 'admin', else 'user'.
  6. userModel document created and saved in MongoDB.
  7. Response: { success: true, message: "User registered successfully" }
  8. Frontend redirects to /login.

LOGIN FLOW:
  1. User fills Login form (email, password).
  2. Frontend POSTs to /login.
  3. Backend special-cases hardcoded admin (admin@gmail.com / admin123)
     → issues JWT immediately if matched.
  4. Otherwise → finds user in DB → bcrypt.compare(password, hashed).
  5. If match → signs JWT: { id, email, role } with 24h expiry.
  6. Response: { success, token, email, fullname, role }
  7. Frontend stores in localStorage: token, email, fullname, role.
  8. If role === 'admin' → navigate to /dashboard.
  9. If role === 'user'  → navigate to /.

AUTHENTICATED REQUEST FLOW:
  1. Axios request interceptor reads token from localStorage.
  2. Adds header: 'x-auth-token': <token>.
  3. Backend auth middleware verifies token signature using JWT_SECRET.
  4. If valid → attaches req.user = { id, email, role } → calls next().
  5. If invalid/missing → 401 response.
  6. Axios response interceptor catches 401 → clears localStorage →
     redirects to /login.

ADMIN REQUEST FLOW:
  1. Same as above but after auth middleware → adminAuth checks req.user.role.
  2. If role !== 'admin' → 403 Forbidden response.

LOGOUT:
  - No dedicated logout endpoint.
  - Frontend clears localStorage (token, email, role, fullname).
  - Redirect to /login.

SECTION 8 — DATA FLOW (DFD)
[Useful for: DFD Level 0, 1, 2]
================================================================================

LEVEL 0 — CONTEXT DIAGRAM:
  [User/Admin Browser] ←──HTTP──→ [Horaistion System] ←──Mongoose──→ [MongoDB]

LEVEL 1 — MAIN PROCESSES:
  Process 1: User Authentication
    Input : Login form data (email, password)
    Output: JWT token + user info → localStorage
    Store : Signup collection (read/write)

  Process 2: Media Browsing
    Input : User navigation (click category/search term)
    Output: Media cards displayed on screen
    Store : media collection (read), mediaMovie collection (read)

  Process 3: Media Detail View
    Input : Media item ID (from URL param)
    Output: Full detail page (synopsis, characters, trailers, reviews)
    Store : media / mediaMovie (read), reviews (read)

  Process 4: Watchlist Management
    Input : User action (add/remove), media info
    Output: Updated watchlist displayed
    Store : Signup collection — watchlist array (read/write)

  Process 5: Review Management
    Input : User comment + rating
    Output: Reviews list updated
    Store : reviews collection (read/write)

  Process 6: Admin Content Management
    Input : Admin form submission (add/edit/delete media)
    Output: Updated media catalog
    Store : media / mediaMovie collections (read/write)

  Process 7: Admin Dashboard Stats
    Input : Admin page load
    Output: Stats cards (totalMedia, totalUsers, totalReviews)
    Store : media, mediaMovie, Signup, reviews (read — countDocuments)

LEVEL 2 — PROCESS 1 (Authentication) DECOMPOSITION:
  1.1 Validate Input    → Check email/password not empty
  1.2 Check Admin Flag  → If admin@gmail.com, skip DB lookup
  1.3 DB Lookup         → userModel.findOne({ email })
  1.4 Password Compare  → bcrypt.compare(input, stored hash)
  1.5 JWT Sign          → jwt.sign({ id, email, role }, secret, 24h)
  1.6 Store Token       → localStorage.setItem('token', ...)
  1.7 Route Redirect    → Navigate by role (admin→/dashboard, user→/)

LEVEL 2 — PROCESS 6 (Admin Content Management) DECOMPOSITION:
  6.1 Add Media
      → Admin fills AddMedia form → POST /addMedia
      → Backend auto-increments id, selects collection by mediaType
      → Saves new Mov or Media document
  6.2 Edit Media
      → Admin clicks Edit on Dashboard → GET /admin/media/:id
      → EditMedia form pre-populated → PUT /admin/media/:id
      → Backend updates Media or Mov by _id
  6.3 Delete Media
      → Admin clicks Delete → DELETE /media/:id
      → Backend tries Media.findByIdAndDelete, then Mov.findByIdAndDelete

SECTION 10 — KEY COMPONENT RESPONSIBILITIES
[Useful for: Component/Class Diagrams, Module Diagrams]
================================================================================

ProtectedRoute.jsx
  - Reads token from localStorage.
  - If no token → <Navigate to="/login" />
  - If adminOnly=true AND role !== 'admin' → <Navigate to="/" />
  - Otherwise renders <Outlet /> (child routes).

Navbar.jsx
  - Displays navigation links.
  - Reads user info (email, role, fullname) from localStorage.
  - Shows admin link if role === 'admin'.
  - Provides search input that navigates to /search?q=...
  - Handles logout (clears localStorage, navigates to /login).

Dashboard.jsx (Admin)
  - On mount: fetches /admin/stats → displays stat cards.
  - Fetches /media → renders media management table.
  - Delete button calls DELETE /media/:id.
  - Edit button navigates to /edit-media/:id.

AddMedia.jsx (Admin)
  - Multi-field form for new media.
  - On submit: POST /addMedia with all form fields.
  - On success: navigates back to /dashboard.

EditMedia.jsx (Admin)
  - On mount: fetches GET /admin/media/:id → pre-fills form.
  - On submit: PUT /admin/media/:id.
  - On success: navigates back to /dashboard.

Animeinfo.jsx (Media Detail Page)
  - Uses URL param :id.
  - Fetches /media (all) → finds matching item by numeric id.
  - Passes data down to children: MediaDetailHero, Information,
    Trailer, Actordata, CommentSection, RecommendationSection.

CommentSection.jsx
  - Fetches GET /reviews/:mediaId on mount.
  - Authenticated users can POST /reviews/add.
  - Authors and admins can DELETE /reviews/:id.

WatchlistPage.jsx
  - Reads email from localStorage.
  - Fetches GET /watchlist/:email.
  - Remove button calls POST /watchlist/remove.

Profile.jsx
  - Reads user info from localStorage.
  - Displays user's watchlist and reviews.
  - Handles POST /change-password.

SearchPage.jsx
  - Reads query from URL (?q=...).
  - Fetches /media (all) → filters client-side by name/genre match.
  - Renders matching MediaCard components.

ErrorBoundary.jsx
  - Class component wrapping entire app.
  - Catches JS errors in render and displays fallback UI.

SkeletonLoader.jsx
  - Placeholder UI shown while async data is loading.

Toast.jsx
  - Renders the toast notification popup.
  - Consumed by ToastContext.

SECTION 11 — SEQUENCE DIAGRAMS (TEXTUAL)
[Useful for: Sequence Diagrams]
================================================================================

--- SEQUENCE: User Logs In ---
Browser → /login page renders Login.jsx
User fills email/password → clicks Submit
Login.jsx → axios.post('/login', { email, password })
  Express /login handler:
    │ Check hardcoded admin shortcut
    │ userModel.findOne({ email })
    │ bcrypt.compare(password, user.password)
    └── jwt.sign({ id, email, role }) → token
Express → Response { success:true, token, email, fullname, role }
Login.jsx → localStorage.setItem('token', ...)
           → localStorage.setItem('role', ...)
           → navigate to /dashboard (admin) or / (user)

--- SEQUENCE: User Adds Item to Watchlist ---
User clicks "Add to Watchlist" on MediaCard or Animeinfo
Component reads email from localStorage
axios.post('/watchlist/add', { email, mediaId, mediaType, name, image })
  [Axios interceptor adds x-auth-token header]
  Express auth middleware verifies JWT
  userModel.findOne({ email })
  user.watchlist.push({ mediaId, mediaType, name, image })
  user.save()
Express → Response { success:true, watchlist: [...] }
Component → shows Toast notification "Added to Watchlist"
WatchlistPage re-fetch updates UI

--- SEQUENCE: Admin Deletes Media ---
Admin on Dashboard clicks Delete button for a media row
Dashboard.jsx → axios.delete('/media/:id')
  [Axios adds x-auth-token header]
  Express adminAuth: verify JWT → check role === 'admin'
  Media.findByIdAndDelete(id) — if null try Mov.findByIdAndDelete(id)
Express → { success:true, message: "Media deleted successfully" }
Dashboard.jsx → removes item from local state → re-renders table

--- SEQUENCE: Admin Edits Media ---
Admin on Dashboard clicks Edit for a row → navigate('/edit-media/:id')
EditMedia.jsx mounts → axios.get('/admin/media/:id')
  Express adminAuth → Media.findById(id) or Mov.findById(id)
  Returns media document JSON
EditMedia.jsx → pre-fills all form fields
Admin edits fields → clicks Save
EditMedia.jsx → axios.put('/admin/media/:id', formData)
  Express adminAuth → Media.findByIdAndUpdate or Mov.findByIdAndUpdate
Express → { success:true, data: updatedDoc }
EditMedia.jsx → navigate('/dashboard')

SECTION 13 — STATE DIAGRAM (User Session)
[Useful for: State Diagrams]
================================================================================

States:
  [Unauthenticated]
    → Event: Successful login
    → Transition to: [Authenticated-User]
    OR
    → Transition to: [Authenticated-Admin] (if role=admin)

  [Authenticated-User]
    → Can navigate all protected user routes
    → Event: Logout or 401 response
    → Transition to: [Unauthenticated]

  [Authenticated-Admin]
    → Can navigate all user routes AND admin routes
    → Event: Logout or 401/403 response
    → Transition to: [Unauthenticated]

  [Token Expired] (sub-state reached from any Authenticated state)
    → Axios interceptor catches 401
    → localStorage.clear()
    → Auto-redirect to /login
    → Transition to: [Unauthenticated]
